# Implementation Plan: Project Bootstrap

**Branch**: `001-project-bootstrap` | **Date**: 2026-05-26 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/001-project-bootstrap/spec.md`

## Summary

Establish the project foundation: Node.js 24 + TypeScript (strict, ESM) REST API with a `/health` endpoint, unified linting and formatting via Biome, 100%-coverage-enforced test suite via Vitest, containerised PostgreSQL service (isolated from the app), and automated git quality gates (pre-commit format check, pre-push test run) via Lefthook. No domain code is introduced.

## Technical Context

**Language/Version**: TypeScript 5.x (strict mode, NodeNext module resolution), Node.js 24

**Primary Dependencies**:
- `fastify` — web framework (schema-first, TypeScript-native)
- `vitest` + `@vitest/coverage-v8` — test runner with V8 native coverage; 100% threshold enforced
- `biome` — unified linting and formatting (single config, single check/fix command)
- `lefthook` — git hooks manager (pnpm-compatible; YAML config)

**Storage**: PostgreSQL (Docker Compose isolated service; no app-level connection at this phase)

**Testing**: Vitest; HTTP route tests via Fastify's `inject()` method; coverage at 100% lines/branches/functions/statements

**Target Platform**: Linux server; local development via Docker Compose

**Project Type**: web-service (REST API)

**Performance Goals**: `/health` responds in < 500ms under normal load

**Constraints**:
- 100% coverage enforced — test run exits non-zero if any line/branch/function is uncovered
- All direct dependencies pinned to exact versions (`.npmrc`: `save-exact=true`)
- Node.js 24 required runtime (`.nvmrc` + `engines` in `package.json`)
- ESM project (`"type": "module"`)
- TypeBox deferred to first domain feature (YAGNI — inline JSON Schema for `/health`)

**Scale/Scope**: Bootstrap phase — one HTTP route, one test file, no domain code

CI and coverage badges are left as placeholders — CI setup is out of scope for this feature.

**Structure Decision**: Hexagonal architecture + vertical slicing (see constitution §Architecture). The `health` slice has no domain or application layer because the health check is pure infrastructure — returning a fixed response requires no business logic. Future domain slices (`restaurants`, `orders`, etc.) will have `domain/`, `application/`, and `infrastructure/` layers. Flat-first rule applies: no subdirectories within `infrastructure/` until there are more than two or three files of the same adapter type. `shared/infrastructure/server.ts` acts as the Fastify app factory, enabling `inject()` in tests without binding to a real port.

## Implementation Order (TDD)

The following sequence enforces Red → Green for each deliverable:

1. **Project skeleton** — `package.json`, `tsconfig.json`, `.npmrc`, `.nvmrc`, `biome.json`, `lefthook.yml`, `docker-compose.yml` — no source files yet
2. **Write `health.test.ts`** (RED) — asserts `GET /health` → 200 `{"status":"ok"}`; fails because route does not exist
3. **Write `app.ts` + `src/health/infrastructure/health.route.ts`** (GREEN) — Fastify factory + route; test passes
4. **Write `src/main.ts`** — entry point (not covered by tests directly; covered via app factory)
5. **Verify coverage** — run `pnpm test`; confirm 100% threshold passes
6. **Verify quality gates** — commit with a formatting violation → rejected; push with a failing test → rejected
